## 🐛 Wolf/Goblin Hallucination Bug (2026-01-04)

**Status**: ✅ FIXED

### Issue Description

The GM LLM was hallucinating monster names that didn't match the actual encounter generated by the system, causing a critical mismatch between what the player sees in the narrative and what exists in the game state.

**Example of the bug:**
```
Encounter System generates: Goblin
GM says: "⚠️ You see: Wolf!"
Player types: "attack the wolf"
Combat happens with: Goblin (causing confusion and broken immersion)
```

### Root Cause Analysis

**The problematic flow:**

1. **Encounter System generates "Goblin"**
   - `encounter.monster_name = "Goblin"`
   - System adds "Goblin" to `npcs_present`

2. **GM receives instruction to describe "Goblin"**
   - Prompt includes: `[ENCOUNTER ALERT: A Goblin (CR 0.25) ...]`

3. **GM LLM ignores instruction and says "Wolf!"**
   - The LLM hallucinated a different monster name in its narrative

4. **Mechanics Extractor reads GM narrative**
   - Sees "Wolf" mentioned in the text
   - Extracts: `{"name": "Wolf", "type": "enemy"}`
   - Adds "Wolf" to `npcs_present`

5. **Both "Wolf" and "Goblin" are now in NPCs list**
   - Player sees "Wolf" in narrative
   - Game state has "Goblin" as the actual encounter
   - Combat system uses "Goblin" for combat mechanics
   - **Result**: Player confusion and broken immersion

### The Fix

Implemented a two-part solution to prevent and mitigate LLM hallucination:

#### Part 1: Filter Hallucinated Enemies

**File**: `dnd_rag_system/systems/gm_dialogue_unified.py` (lines 549-575)

When an encounter is active, filter out any enemy NPCs extracted from the narrative that don't match the actual encounter monster:

```python
# After extracting mechanics from GM response
if encounter and mechanics.npcs_introduced:
    filtered_npcs = []
    
    for npc in mechanics.npcs_introduced:
        npc_name = npc.get('name', '').strip().lower()
        encounter_name = encounter.monster_name.strip().lower()
        
        # Allow the actual encounter monster
        if npc_name == encounter_name:
            filtered_npcs.append(npc)
        # Allow non-enemy NPCs (merchants, guards, etc.)
        elif npc.get('type') != 'enemy':
            filtered_npcs.append(npc)
        else:
            # This is a hallucinated enemy - filter it out
            logger.debug(f"Filtered hallucinated enemy '{npc_name}'")
    
    mechanics.npcs_introduced = filtered_npcs
```

**What this does**:
- ✅ Keeps the correct encounter monster (Goblin)
- ✅ Keeps friendly/neutral NPCs (Merchant, Guard, etc.)
- ❌ Filters out hallucinated enemy monsters (Wolf, Dragon, etc.)

#### Part 2: Strengthen Encounter Instructions

**File**: `dnd_rag_system/systems/encounter_system.py` (lines 259-280)

Made the encounter instruction much more prominent and explicit:

```python
def format_encounter_for_gm(self, encounter: EncounterResult) -> str:
    return (
        f"\n═══════════════════════════════════════════════════════════════════\n"
        f"🎲 MANDATORY ENCOUNTER - USE THIS EXACT MONSTER NAME 🎲\n"
        f"═══════════════════════════════════════════════════════════════════\n"
        f"A {encounter.monster_name} (CR {encounter.monster_cr}) is nearby!\n"
        f"CRITICAL: You MUST say '{encounter.monster_name}' (not any other creature).\n"
        f"Describe it appearing, approaching, or being spotted by the player.\n"
        f"═══════════════════════════════════════════════════════════════════\n"
    )
```

**Changes made**:
- Added visual separators (borders) to make instruction stand out
- Used "MANDATORY" and "CRITICAL" keywords for emphasis
- Explicitly told the LLM to use the exact monster name
- Warned against using other creature names

## 🐛 Selenium HP Extraction Bug (2026-01-04)

**Status**: ✅ FIXED

### Issue Description

Selenium E2E tests were failing with the error:
```
Unable to locate element: {"method":"css selector","selector":"textarea[label="Character Sheet"]"}
```

The tests couldn't extract character HP from the UI, causing test failures and displaying "HP: ???" in test output.

### Root Cause Analysis

The test was using an outdated selector that assumed the character sheet was rendered as a textarea:

```python
# OLD (broken):
sheet = driver.find_element(By.CSS_SELECTOR, 'textarea[label="Character Sheet"]')
text = sheet.get_attribute('value')
```

**Why it broke:**
- The UI component `character_sheet` in `web/app_gradio.py` (line 1058) is a `gr.Markdown()` component
- Gradio renders Markdown as HTML elements (`<div>`, `<p>`, `<strong>`), not `<textarea>`
- The actual HTML structure is: `<strong>HP</strong>: 28/28`
- The CSS selector was looking for an element that never existed in the current UI

### The Fix

**File**: `e2e_tests/test_adventure_simulation.py` (lines 70-93)

Updated the HP extraction function to:
1. Search for HP text in any element (not just textarea)
2. Use regex to parse the HP pattern
3. Fallback to searching page source if needed

```python
def get_hp_from_sheet(driver):
    """Extract HP from character sheet."""
    try:
        # Look for HP text in the page (it's in markdown/div, not textarea)
        # HP appears as: <strong>HP</strong>: 28/28
        elements = driver.find_elements(By.XPATH, "//*[contains(text(), 'HP')]")
        
        for elem in elements:
            text = elem.text
            if 'HP:' in text or 'HP' in text:
                # Try to find pattern like "HP: 28/28" or "HP 28/28"
                import re
                match = re.search(r'HP[:\s]+(\d+)/(\d+)', text)
                if match:
                    current_hp = int(match.group(1))
                    return current_hp
        
        # Fallback: search page source
        page_source = driver.page_source
        match = re.search(r'HP[:\s]+(\d+)/(\d+)', page_source)
        if match:
            return int(match.group(1))
            
    except Exception as e:
        print(f"⚠️  Could not parse HP: {e}")
    return None
```
